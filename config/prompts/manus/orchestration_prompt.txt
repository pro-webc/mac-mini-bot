以下の手順を順に実行せよ。停止・質問・サボりは禁止。無限ループを回避し正確に実装すること。大規模な作業になってしまう場合は、同時に作業するのではなく、一つずつ確実に実装するようにしてください。

**【最重要 — ソースコードの取得方法（これを間違えると全工程やり直し）】**
リファクタリング元のソースコードは **このプロンプトの末尾** `===== BEGIN_CANVAS_SOURCE =====` ～ `===== END_CANVAS_SOURCE =====` に **全文が埋め込み済み** です。
- **リポジトリ内の `canvas_code.txt` は空のテンプレートファイルであり、ソースコードではありません。`canvas_code.txt` を読みに行かないでください。**
- **Canvas やブラウザでの外部取得も一切不要です。**
- 手順 2 のリファクタリングでは、**プロンプト末尾の埋め込みコード**だけを入力として使用すること。

【表現】ソース・コミットメッセージ・作業メモに **装飾用の Unicode 絵文字を含めない**。ユーザー向け UI 文言にも **原稿に無い絵文字を追加しない**。

【手順1: リポジトリ作成】
propagate-webcreationのワークスペースにリポジトリを新たに作成してください。
- **リポジトリ名**は次の文字列と**完全一致**させること: `{{MANUS_REPO_NAME}}`
  （フォーマットは「レコード番号-ASCII部分」。GitHub はリポジトリ名に日本語を受け付けないため、パートナー名の英数字部分のみを使用。ASCII 部分が無い場合はレコード番号のみ。**この文字列を1文字も変えないこと**）
- **ディスクリプション**には次の1行を**そのまま**入力すること: `{{MANUS_REPO_DESCRIPTION}}`
  （「レコード番号 パートナー名」形式。制作スプレッドシートの値そのまま。工程テスト・本番を問わずこの形式）
privateのままで、テンプレートリポジトリとしてpropagate-webcreation/DefaultSettingを使用してリポジトリを作成してください。
※もし、作成しようとしたリポジトリ名が既にあるなら、**作業を直ちに中止し、エラーとして報告**してください。既存リポジトリの削除・上書きは禁止です。「リポジトリ `{{MANUS_REPO_NAME}}` は既に存在するため作成できません。手動で確認してください。」と回答して終了すること。

【手順2: リファクタリング】
作成したリポジトリをクローンして、**このプロンプトの末尾に埋め込まれた**「リファクタリング元のソースコード」を以下の「リファクタリング指示書」に従ってリファクタリングを実装してください。

**【ページ数の厳守（最重要 — ビルド検証で自動的に失敗する）】**
本プロンプト後半の **【契約ページ数・法的情報】** に記載された `page.tsx` の本数を **1本たりとも超過してはならない**。元コードに存在しないページ（blog, privacy, terms 等）の `page.tsx` を新規作成することは禁止。元コードのページ構成をそのまま App Router のルーティングに変換し、契約本数と完全に一致させること。超過するとパイプラインのビルドチェックで自動的に失敗し、案件全体がやり直しになる。

**【ソースコード取得に関する厳守事項 — 違反は全工程やり直し】**
- リファクタリング元のソースコードは、**このプロンプト末尾**の `===== BEGIN_CANVAS_SOURCE =====` ～ `===== END_CANVAS_SOURCE =====` ブロック内に**全文が埋め込み済み**です。外部から取得する必要はありません。
- **禁止行為（以下を行うと配色・レイアウトが崩壊し全工程やり直しになる）:**
  - リポジトリ内の `canvas_code.txt` を読みに行くこと（空のテンプレートファイルであり、ソースではない）
  - `blog.md` 等のテンプレートファイルからコードを取得しようとすること
  - Canvas・外部チャット・ブラウザへのアクセスやサインイン
  - プロンプト末尾の埋め込みコードを無視してスクラッチで独自実装すること
- 手順: (1) リポジトリをクローン → (2) **このプロンプト末尾の `BEGIN_CANVAS_SOURCE` ～ `END_CANVAS_SOURCE` 内のコードを読み取る** → (3) そのコードをリファクタリング指示書に従って実装する
- **自己検証**: クローン後に `canvas_code.txt` が空であっても問題ありません。ソースはプロンプト末尾にあります。`canvas_code.txt` が空であることを理由にスクラッチ実装に切り替えてはなりません。

**【コンテンツ量の維持 — 最重要ルール・違反は即不合格】**
リファクタリングとは「コードの構造変換」であり「内容の編集・要約」ではない。元のソースコードに含まれる**すべてのユーザー向けテキストを一字一句変更・省略・要約しない**。コンポーネント分割のために文量を減らさない。
**よくある違反例（以下はすべて禁止）:**
- 会社概要テーブルの行を減らす（代表者名・完全な住所・設立年月・メールアドレス・営業時間等の欠落）
- 料金・価格情報を省く
- サブサービスやサブプロジェクトの個別説明を1つにまとめる・要約する
- キャンプ日程やスケジュールの具体的な日付・場所を省く
- 特典・キャンペーン情報を省く
- 口コミ・お客様の声・実績の件数を減らす
- FAQの質問・回答の数を減らす
- 社会的証明（数値データ・統計）を省く
- 「選ばれる理由」等の箇条書き項目を減らす

**【レイアウト構造の維持 — テキストと同等に最重要・違反は即不合格】**
リファクタリングとは「ファイル構造の変換」であり「UIデザインの再設計」ではない。元コードの**レイアウト構造・セクション構成・コンポーネントの表示パターン**を勝手に変更してはならない。
**よくある違反例（以下はすべて禁止）:**
- ヒーローセクションの**背景画像+オーバーレイ構造**をCSSグラデーションに差し替える
- ヒーローの**全幅背景画像+中央テキスト**レイアウトを**2カラムグリッド（テキスト左/画像右）**に変更する
- 下層ページヒーローの**固定高さ（h-[400px] 等）**をpadding ベースに縮小する
- コンポーネントの**propsを勝手にリネーム・削除**する（例: `catchCopy` を消して `title` だけにする）
- 元コードにあるキャッチコピー（例: 「SNSを、成果に変える」）をリファクタ後に落とす
- 元コードの**グリッド列数・カラム比率**を変更する
- `absolute inset-0` による背景レイヤー構造を `relative` の横並び配置に変更する
- 元コードが**1カラム中央揃え**のセクションを**2カラム**に変更する（またはその逆）
**自己チェック**: リファクタ前後で各セクションの「レイアウトパターン」（全幅背景/グリッド/カード並び/中央揃え等）が同一であることを確認する。

**【実装前の必須作業: コンテンツ棚卸し】**
リファクタリング実装を開始する**前に**、元ソースコードの全セクション（見出し単位）を列挙し、以下を記録せよ:
- テキスト量（段落数・表の行数・FAQ数・カード枚数・リスト項目数）
- **レイアウトパターン**（全幅背景/2カラムグリッド/カード列/中央揃えテキスト等）
- **背景の実装方式**（CSS gradient / bg-[url(...)] 背景画像 / 単色 / 画像オーバーレイ等）
- **propsの一覧**（コンポーネントが受け取る props 名と型）

**【実装後の必須作業: コンテンツ・レイアウト監査】**
全コンポーネントの実装完了後、**ビルドの前に**棚卸しリストと突合して以下を確認せよ:
1. 元ソースの全セクションがリファクタ後にも存在する
2. 各セクション内のテキスト段落数・表の行数・FAQ数・カード枚数・箇条書き項目数が一致する
3. 料金・日程・住所・メール等の具体的な数値・固有名詞がすべて残っている
4. **各セクションのレイアウトパターンが元コードと同一である**（背景画像ありのセクションが gradient だけになっていないか等）
5. **コンポーネントの props が元コードと同一である**（props のリネーム・削除がないか）
6. **元コードで定義されたカスタムフック・ユーティリティが全て import され使用されている**（作成したが未配線のファイルがないか）
不一致がある場合は**ビルドに進まず、元コードと一致するよう修正**すること。

元のソースコードは単一ファイルですが、複数ページ（HomePage, AestheticPage, ImplantPage等）を内包している可能性があるので確認してください。シングルページの場合もあります。
各ページをNext.jsのルート（/, /aesthetic, /implant等）として分離し、さらに各ページ内のセクションを個別のコンポーネントファイルに分割してください。
各page.tsxでは、そのページ用のセクションコンポーネントをimportして配置するだけにしてください。

完了前の必須アクション
【CSS実装時の必須ルール】
globals.cssやCSSファイルを作成・編集する際は、以下のルールを厳守してください：
1. **@importルールの配置:** 必ず@importルールをファイルの先頭（コメントの後）に配置すること
2. **順序の厳守:** :root変数、@theme、その他のCSSルールは必ず@importの後に記述すること
3. **外部フォントの読み込み:** Google Fontsなどの外部フォントはCSS @importではなく、layout.tsxのHTML <link>タグで読み込むこと
4. **正しいCSS順序テンプレート:**
```css
/* コメント */
@import "tailwindcss";
:root { /* CSS変数 */ }
@theme { /* Tailwind設定 */ }
/* その他のCSS */
```
実装完了後、指示書の「追加の絶対制約」に違反していないか必ず自己レビューを行い、修正してから次の手順に進んでください。

【手順3: 画像生成と実装】
コード内の`ImagePlaceholder`等のtext/subText、プレースホルダーやサイト全体のトーンを分析し、nanobananaProで画像を生成・実装する。

■人物描写ルール
* **基本ルール:** 必ず「日本人（Japanese）」を指定してください。
* プロンプト例: `Japanese professional`, `Japanese woman`, `Japanese office environment`

■プロセス
1. **分析とプロンプト作成**
**画像プレースホルダーの特定:**
- ImagePlaceholder系コンポーネント（名称問わず）
- text/imgText等のprops、aspect-ratio/heightClass等の画像領域
- コンポーネント内の動的生成（map/loop/配列データ）、子コンポーネント内も含む
- **背景画像として使用されている箇所**（`bg-[url(...)]`・`style={{ backgroundImage }}` 等）
- **ヒーローセクション等のオーバーレイ付き背景画像**（`absolute inset-0` + 画像 + グラデーションオーバーレイの構造）
**抽出手順:** 元のソースコード全体から上記を全て抽出し、漏れがないことを確認。
以下のJSON形式で出力。promptに"no text"必須。**全プレースホルダー分析必須。**
```json
[
  {
    "id": "ファイル名",
    "prompt": "(英語プロンプト。人物描写のルールを適用すること。高品質な呪文も含む)",
    "negative_prompt": "low quality, text, watermark, bad anatomy",
    "aspect_ratio": "(16:9 または 4:3 など)",
    "usage": "(foreground-image | background-image | card-image 等、用途を明記)",
    "reason": "選定理由"
  }
]
```

2. **画像生成 (Step 1)**
JSONに基づきnanobananaProで生成。文字混入厳禁。必ず全てのプロンプトで画像を作成すること。
画像保存: 生成した画像は /public/images/ ディレクトリに保存してください（ディレクトリがない場合は作成すること）。
【必須】文字の完全除外: 生成する画像に文字、テキスト、レター、数字、記号などの書かれた文字を一切含めないこと
## 画像生成時の必須チェックリスト
- [ ] 文字・テキストを一切含めない
- [ ] 人物は必ず「Japanese」を指定

3. **実装 (Step 2)**
- 保存先: `/public/images/` (無ければ作成)
- パス置換: ソース内の外部URL等をローカルパス(`/images/xxx`)に置換。`src`に`/public`は含めない。
- **背景画像の実装**: 元コードで `bg-[url(...)]` や `style={{ backgroundImage }}` を使っている箇所は、**同じ CSS 手法で**生成画像のパスに差し替える。`next/image` の `<Image fill>` に変換してもよいが、その場合も **`absolute inset-0` + `object-cover` + z-index でオーバーレイを維持**し、元コードと同じ視覚的効果を保つこと。**背景画像をサイド配置の `<Image>` に変更することは禁止**。
- **パス監査（必須）**: コミット前に全 `<Image` の `src` を列挙し、(1) `/public` を含む誤記が無いこと (2) ローカルパスなら `public/` 下に実ファイルがあること を確認。欠落は画像追加または SVG で補完してからビルドする。
- コンポーネント: `next/image`から`Image`をインポートして使用。ローカル画像をパス文字列で指定する場合、<Image /> コンポーネントには必ず width と height プロパティを指定するか、fill プロパティ（親要素に relative 必須）を使用してください。これを守らないとビルドエラーになります。
既存のクラス名（Tailwind CSS）は維持し、デザインが崩れないようにしてください。
**画像実装の完全性確認（必須）:**
1. 元のソースコードで画像が表示されていた箇所を全てリストアップ
2. 生成した画像の数と、元のソースコードの画像表示箇所の数が一致することを確認
3. 不一致の場合は、漏れている箇所を特定し、追加で画像生成を実施
4. 実装後、全ページの全セクションで画像が正しく表示されていることを確認
5. 画像プレースホルダーが残っている箇所がないことを確認
6. **生成した全画像が実際にコードから参照されていることを確認**（生成したが `src` に設定されていない画像がないか）
7. **背景画像として使われるべき画像が、背景として実装されていることを確認**（`hero-bg` 等の名前で生成した画像がサイド画像や未使用になっていないか）

4. **CSS アニメーション動作確認（必須）**
- globals.css に `opacity: 0` で始まるアニメーションクラス（`.reveal`・`.fade-in` 等）がある場合、**それらを `opacity: 1` に変更する JavaScript（IntersectionObserver 等）が実装され、layout.tsx またはコンポーネントから実際に import・実行されていることを確認する**
- **カスタムフックやユーティリティを作成した場合、それが実際にコンポーネントから import され使用されていることを確認する**（ファイルだけ作って未配線は禁止）
- `opacity: 0` のままユーザーに見えないセクションが無いことを、ビルド後のページで確認する

5. **検証とプッシュ (Step 3)**
- `npm run build`を実行。
- エラー時は人間に聞かずログを読み、成功するまで「修正→ビルド」を繰り返す(自己修復)。
- ビルド完全成功後にプッシュする。

モデルは **Manus 1.6** を使用すること（手作業マニュアルどおり）。
