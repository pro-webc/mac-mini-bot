# Next.js（App Router）+ Tailwind CSS リファクタリング指示書

## 前提（最重要 — 必ず最初に確認。違反は全工程やり直し）

- リファクタリング元のソースコードは、**このプロンプト末尾**の `===== BEGIN_CANVAS_SOURCE =====` から `===== END_CANVAS_SOURCE =====` までに**本文として埋め込み済み**（外部ファイルの参照・添付ではない。Canvas やブラウザでの取得も一切不要）
- 上記ブロック内の **TSX コードフェンス**をリファクタリング元のソースコードとして扱う
- **`canvas_code.txt` は絶対に読みに行かないこと**: テンプレートリポジトリ内の `canvas_code.txt` / `blog.md` 等は**空の雛形ファイルであり、ソースコードの格納場所ではない**。`canvas_code.txt` が空であっても正常。空だったことを理由にスクラッチ実装に切り替えることは禁止
- 既存の **Next.js（App Router）+ Tailwind CSS の空テンプレート**に統合する
- **最優先事項**
  - デザイン
  - レイアウト
  - 配色
  - テキスト内容

👉 **元コードを 100% 維持すること（変更不可）**

### 絵文字・装飾記号（必須）

- リファクタ後の **ユーザー向けテキスト**（見出し・本文・ボタン・ナビ・FAQ・プレースホルダ）に **Unicode 絵文字を新規で追加しない**。原稿・Canvas に明示的に含まれるものだけ**そのまま**残す。
- `aria-label`・`alt`・コミットメッセージに **装飾用の絵文字を入れない**。
- 説明や作業ログの文章にも **絵文字を使わない**（プレーンテキスト）。

### 表示上の例外（構成案メタの除去のみ）

- 画面上に見えている文字のうち **wireframe 用メタ**（例:「【画像・配置の指示】」の長文、接頭辞「【画像プレースホルダー】」、制作指示だけの段落）**のみ**を、**レイアウト・余白・配色の意図を崩さない範囲**で除去するか JSX コメントに移してよい。
- **見出し・リード・本文・表・ボタン文言・FAQ・価格など、ユーザー向け原稿は変更しない**（コピーの差し替え・短縮・追加は禁止）。
- 除去後も画像エリアは**短いプレースホルダ**（数文字〜1 短句）または実画像で埋め、**プロ品質の余白・整列・タイポ階層**を維持すること。

### レイアウト構造の維持（デザイン維持と同等に最重要）

リファクタリングは**ファイル分割のみ**の作業であり、**UIレイアウトの再設計は一切行わない**。

- **セクションのレイアウトパターンを変更しない**: 元コードが「全幅背景画像+オーバーレイ+中央テキスト」なら、リファクタ後も同じ構造にする。「2カラムグリッド（テキスト左/画像右）」に変更することは禁止。
- **背景画像の実装方式を変更しない**: 元コードが `bg-[url(...)]` や `absolute inset-0` + `<img>` で背景画像を敷いている場合、CSSグラデーションだけに置き換えてはならない。
- **セクションの高さ・サイズ指定を変更しない**: 元コードが `h-[550px] md:h-[750px]` のような固定高さなら、リファクタ後も同じ値を維持する。padding ベースの高さに縮小することは禁止。
- **コンポーネントの props を勝手にリネーム・削除・追加しない**: 元コードの関数コンポーネントが受け取る props（例: `title`, `catchCopy`, `subtitle`）をリファクタ後も全て維持する。props名の変更（例: `catchCopy` → `description`）や削除（`catchCopy` を落とす）は禁止。
- **グリッド列数・カラム比率を変更しない**: 元コードが `grid-cols-3` なら `grid-cols-2` にしない。`md:grid-cols-2` を `md:grid-cols-1` にしない。
- **配置方式を変更しない**: `absolute inset-0` による背景レイヤー構造を `relative` の横並びに変更しない。`justify-center` を `justify-start` に変更しない。

### レイアウト・UI の微調整（許可範囲）

- **構成案や Canvas のワイヤ指示にピクセル単位で合わせる必要はない**。コンポーネント分割に伴い、**margin・padding の微調整**（±数px/rem 程度）は許可する。ただし上記の「レイアウト構造の維持」の範囲を超える変更は禁止。
- **カード・ボタンの見た目の磨き込み（追加可）**: Canvas に無くても、**Tailwind ユーティリティのみ**で **`shadow-sm`〜`md` または `ring-1 ring-black/5`**、**`hover:-translate-y-0.5` + `hover:shadow-md`**、**`transition-all duration-200`〜`300`**、**`active:scale-[0.98]`**（ボタン）を付与してよい。**HEX・文言・項目数は Canvas と同一**。**`shadow-2xl` 級の濃影の乱発・点滅・自動再生**は禁止。`prefers-reduced-motion` 配慮は共通技術要件に従う。
- **文言・数値・CTA ラベル・カラーコード**は Canvas と同一を維持する（勝手な色変更やコピー改変は禁止）。
- **リファクタで段落・箇条・FAQ を統合・要約してページを薄くしない**。Canvas 上の**文量・項目数**は維持する（メタ除去のみ可）。

### コンテンツ量の完全維持（最重要 — 違反は即不合格）

- Canvas 上の**テキスト本文**（見出し・リード・本文・表セル・FAQ・口コミ・料金表・スケジュール表・プロフィール・会社概要・特典・実績・選ばれる理由）の**文量をリファクタで減らすことは禁止**。
- レイアウトの都合でテキストを短くする・項目を統合する・行を削除する等の「見た目のための要約」は行わない。
- 元コードに存在するセクション・カード・リストアイテムの**数を減らさない**。
- リファクタリングとは「元コードのJSXテキストを新しいファイル構造にコピーして配置する作業」である。**自分の言葉で書き直してはならない**。元コードのテキストを**そのままコピー**すること。
- **新しいページコンポーネントを作成するとき**: 元コードの該当ページのJSXを**丸ごとコピーしてから**コンポーネントに分割する。記憶に頼って短い文章を書くのではなく、元コードを**直接参照してコピー**すること。

#### コンテンツ・レイアウト監査チェックリスト（ビルド前に必ず実行）

リファクタリング完了後、以下を**1項目ずつ**確認し、不一致があれば修正してからビルドに進む:

- [ ] 元コードの全ページ（トップ・サービス詳細・会社概要・お問い合わせ等）がリファクタ後にも存在する
- [ ] 各ページ内の**セクション数**が元コードと一致する
- [ ] **テーブル（表）の行数**が元コードと一致する（会社概要の項目数を含む）
- [ ] **FAQ の質問・回答ペア数**が元コードと一致する
- [ ] **カード・リスト項目の数**が元コードと一致する
- [ ] **料金・価格**がすべて記載されている
- [ ] **日程・スケジュール**の具体的な日付がすべて記載されている
- [ ] **特典・キャンペーン**情報が記載されている
- [ ] **口コミ・お客様の声**の件数が元コードと一致する
- [ ] **数値データ**（統計・実績の数字）がすべて記載されている
- [ ] **各セクションのレイアウトパターン**が元コードと同一（全幅背景 / 2カラム / カード列 / 中央テキスト等）
- [ ] **ヒーローセクションの背景実装**が元コードと同一（背景画像なら背景画像のまま。gradient のみへの差替え禁止）
- [ ] **コンポーネントの props 名・数**が元コードの設計と同一（リネーム・削除なし）
- [ ] **元コードのカスタムフック・ユーティリティが全て import され使用されている**（作成したが未配線のファイルなし）
- [ ] **`opacity: 0` で開始するアニメーションCSS**がある場合、それを `opacity: 1` にする JavaScript（IntersectionObserver 等）が実装・接続されている
- [ ] **生成した全画像が実際にコードの `src` から参照されている**（生成したが未使用の画像なし）

## 1. プロジェクト解析とルーティング決定

### 解析ルール
末尾のリファクタリング元ソースコードの構造を解析し、以下 **A / B のどちらか**を判定する。

**契約ページ数の優先（必須）:** 同一プロンプト内の **【契約ページ数・法的情報】**（本リファクタ指示書の直後に付与される）に契約ページ数が **2 以上** とある場合は、上記 A/B の見た目上の判定より **契約を優先**する。元 Canvas が `useState` 無しの縦長1ファイルでも、**その本数の `page.tsx` に必ず分割**し、Pattern A に落とさないこと。

---

### パターンA：シングルページ構成（LP型）

**特徴**
- `useState` 等による画面切り替えが存在しない
- スクロールでコンテンツを見せる構成

**実装方針**
- ルート: `app/page.tsx`
- セクション単位でコンポーネント分割

**構成例**
```text
app/components/home/
├── HomeHero.tsx
├── HomeServices.tsx
├── HomeFeatures.tsx
└── HomeContact.tsx
```

`app/page.tsx` では、上から順にコンポーネントを並べるのみ。

---

### パターンB：複数ページ構成（擬似SPA）

**特徴**
- `useState` による条件分岐で Home / About / Contact 等を切り替えている

**実装方針**
- Next.js のファイルベースルーティングに変換
- 状態管理による切り替えは禁止
- ページ単位、セクション単位でコンポーネント分割

**構成例**
```text
app/
├── page.tsx          # Home
├── about/page.tsx    # About
└── contact/page.tsx  # Contact
```

コンポーネントはページごと、セクションごとに分離する。

### 契約ページ数と法的情報（必須 — 違反は即不合格）

- 契約で定めた **コンテンツ用 `app/**/page.tsx` の本数は、契約ページ数と完全に一致**させる（超過も不足も禁止。後段の【契約ページ数・法的情報】の数値に従う）。
- **元コードに存在しないページの `page.tsx` を新規作成することは禁止**。元コードのページ構成をそのまま App Router のルーティングに変換すること。`blog/page.tsx`・`news/page.tsx`・`privacy/page.tsx`・`terms/page.tsx` 等を元コードに対応するセクションがないのに追加してはならない。
- プライバシーポリシー・利用規約等は **契約ページ数に含めない**。**新規の `app/**/page.tsx`（例: `privacy/page.tsx`）を作らない**。
- これらは **既存のいずれかのページ上**で `Dialog` / `Sheet` / `Drawer` / `<details>` 等により全文表示する。フッターからは **同一ページ内の UI** で開く（`/privacy` 等へのナビ導線を新設しない）。

## 2. ディレクトリ構造ルール（Strict）

### 基本ルール(絶対反映)
- **すべてのコンポーネントは `app/components` 配下**
- `page.tsx` は「組み立て専用」
- JSX やロジックは極力コンポーネント側に記述

---

### コンポーネント構成例
```text
app/components/
├── home/                  # トップページ用
│   ├── HomeHero.tsx
│   ├── HomeFeatures.tsx
│   └── HomePricing.tsx
├── about/                 # Aboutページ用
│   └── AboutContent.tsx
└── shared/                # 共通パーツ
    ├── Header.tsx
    ├── Footer.tsx
    ├── Button.tsx
    └── LayoutWrapper.tsx
```

---

### ページファイルの役割

- `app/**/page.tsx`
  - コンポーネントを import して並べるだけ
  - ロジック・詳細な JSX を書かない

## 3. 実装要件（Strict Requirements）

### Client / Server Component

- `app/**/page.tsx`
  - **原則 Server Component**
- 以下を使用する場合は **必ず Client Component**
  - `useState`
  - `useEffect`
  - イベントハンドラ
- Client Component の先頭に必ず記述：
```ts
"use client";
```

---

### 画像処理

- `<img>` タグは禁止
  → **`next/image` の `<Image />` に置換**
- **`src` の書き方**: ローカル静的ファイルは **`/images/ファイル名`**（`public/images/` に置いたファイルは URL は `/images/...`）。**`src` に `/public/` や `public/` を含めない**（表示されない典型原因）。
- **寸法**: ローカル `src` では **`width` と `height`（数値）**、または **`fill`**（その場合は親に **`relative`** と、高さまたは **`aspect-*` で表示領域を確定**）。欠けるとビルドエラーになり得る。
- **実在チェック**: コミット前に `<Image` の `src` を列挙し、ローカルパスなら **`public/` 下にファイルが存在する**ことを確認する。`ImagePlaceholder`・`src="#"`・存在しないパスを残さない。
- 外部 URL を使う場合のみ `next.config.ts` の **`images.remotePatterns`（等）** を追加し、ビルドで検証する（原則はローカル `public/images/`）。

### 背景画像の処理（特に重要）

元コードで**背景画像**として実装されている箇所（ヒーローセクション等）は、リファクタ後も**背景画像として維持**する。

**許可される実装パターン:**
- `next/image` + `fill` + `absolute inset-0 object-cover` + z-index によるレイヤー構造
- CSS `background-image` + `background-size: cover`

**禁止される変更:**
- 背景画像を削除して CSS gradient のみにする
- 背景画像をサイド配置の前景 `<Image>` に変更する
- オーバーレイ（暗化・グラデーション）を削除する

---

### リンク

- `<a>` や `onClick` によるページ遷移は禁止
- 必ず `next/link` の `<Link />` を使用

---

### スタイリング

- **Tailwind CSS のクラスは一切変更しない**（`bg-[#XXXXXX]` / `text-[#XXXXXX]` を含む全てのユーティリティクラス）
- 元コードにある以下の定数は切り出しを検討
  - COLORS
  - FONTS
- 移動先
  - `tailwind.config.ts`
  - または定数ファイル（例: `app/constants/theme.ts`）
- **定数化の際もカラーコードの値を変更しない**。`bg-[#2360E5]` → `bg-primary` のようにエイリアスする場合でも、`primary` の値は `#2360E5` と完全一致させる。

---

### Global CSS に関する厳格ルール

- `app/globals.css` には **何も記述しない**
- Tailwind のユーティリティクラスのみでスタイリングを完結させる
- 独自の CSS 定義・リセット・上書きは禁止

## 6. Global CSS の例外ルール（重要）

### 原則（既存ルールを維持）
- `app/globals.css` には **何も記述しない**（独自CSS・リセット・上書きは禁止）
- Tailwind ユーティリティでスタイリングを完結させる

### 例外（唯一許可）
- 元コード内に **`<style>` タグでカスタムCSSが存在する場合のみ**、その内容を **`app/globals.css` に移植してよい**

### 移植が許可されるもの（例）
- `.hero-bg`, `.btn-3d`, `.section-title::after` などの **カスタムクラス**
- `::before / ::after` など **疑似要素**
- Tailwind だけでは再現困難な **複雑なグラデーション / マスク / filter / keyframes** 等
- **スクロールアニメーション用の CSS**（`.reveal { opacity: 0; transform: ... }` 等）

### 判断基準（厳守）
- 元コードの `<style>` が **空**なら、`app/globals.css` も **空（最小）**
- 元コードに存在しないCSSを **新規に追加することは禁止**
- 移植は **"そのままコピー"**が原則（意味を変える編集禁止）

### CSS アニメーション初期状態の動作保証（必須）

**`opacity: 0` で始まる CSS アニメーションクラス**（`.reveal`, `.fade-in`, `.scroll-animate` 等）を globals.css に移植した場合、**それを `opacity: 1` に変更する JavaScript が実装・接続されていないと、ユーザーにはコンテンツが見えなくなる**。

- 元コードに IntersectionObserver や scroll event listener がある場合は、**Client Component として実装し、layout.tsx から import して配置する**
- カスタムフック（例: `useScrollReveal`）として実装する場合は、**それを使用する全コンポーネントで実際に import して `ref` を渡す**。ファイルを作っただけで使わないことは禁止
- **代替策**: scroll animation の JavaScript が複雑な場合、CSS に `animation-timeline: view()` を使うか、`opacity: 0` の代わりに **初期状態を `opacity: 1` にして、アニメーションは追加効果のみ**にする

## 4. Hydration Error 防止（最重要）

### HTML ネストルール（厳守）

❌ 禁止例
```html
<p>
  <div>...</div>
</p>
```

- `<p>` タグの中に以下を **絶対に入れない**
  - `<div>`
  - `<section>`
  - `<ul>`
  - `<h1>`〜`<h6>`

---

### 動的要素の扱い

- 日時・ランダム値などは以下のいずれかで対応
  - `useEffect` でマウント後に描画
  - `suppressHydrationWarning` を付与

---

### layout.tsx 設定

```tsx
<body suppressHydrationWarning={true}>
```

- ブラウザ拡張機能対策として必須

---

### window / document

- 直接アクセスは禁止
- 必ず以下の中で使用
  - `useEffect`
  - イベントハンドラ内

---

## 追加の絶対制約（違反したら不合格）

### 配色の完全維持（最優先・違反は即不合格）

元コードで指定されている **background-color / color / bg-[...] / text-[...]** のカラーコードは **1つも変更してはならない**。

- 元コードの `<body>` や `<main>` に `background-color: #XXXXXX` や `bg-[#XXXXXX]` が設定されている場合、**リファクタ後も同じ値を維持する**。白背景（`#FFFFFF` / `bg-white`）への変更は禁止。
- セクションごとの背景色（`bg-[#2360E5]` / `bg-[#1A4BB8]` 等）を削除・変更しない。
- テキスト色（`text-[#FFFFFF]` / `text-white` 等）を `text-gray-*` / `text-black` 等に変更しない。
- テンプレートのデフォルトスタイル（白背景・暗テキスト）で元コードの配色を上書きしない。テンプレート側の `globals.css` や `layout.tsx` のデフォルト配色と元コードの配色が競合する場合は、**元コードの配色を優先する**。
- **自己チェック**: リファクタ完了後、元コード内のすべての HEX カラーコード（`#` で始まる値）が出力にも同一の形で含まれることを確認する。欠落があれば修正してからビルドに進む。

### A. page.tsx 禁止事項（強制）
- app/**/page.tsx 内に Tailwind className を1つでも書くことを禁止する。
- app/**/page.tsx 内に div/section/header/footer/main 等のレイアウト要素を書くことを禁止する。
- app/**/page.tsx に残して良いのは以下のみ：
  1) import 文
  2) export default function Page()
  3) <Component /> を上から順に並べる記述（ラッパーdivすら禁止）
  4) metadata / generateMetadata（必要な場合のみ）

→ つまり page.tsx は「組み立て専用の薄いファイル」以外の存在を許さない。

### B. JSX の移動ルール（強制）
- 元コードの JSX は "100%" app/components/** に移すこと。
- page.tsx に JSX を残すことは「デザイン維持のため」でも例外なく禁止。
- 例外として許可するのは <Fragment> と <Component /> のみ。

### C. 出力物の定義（強制）
- 最終出力には必ず以下を含める：
  - app/page.tsx（上記Aを満たす）
  - app/components/** の全ファイル（元の JSX を完全移植）
  - もし Header/Footer が存在するなら app/components/shared に配置

### D. カスタムフック・ユーティリティの配線義務（強制）
- app/components/** にカスタムフック（`use*.ts`）やユーティリティ（`*.ts`）を作成した場合、**そのファイルを import して使用するコンポーネントが必ず1つ以上存在すること**
- **ファイルを作成したが、どのコンポーネントからも import されていない**状態は禁止
- ビルド前に `app/components/**/*.ts` を列挙し、各ファイルが他のファイルから import されていることを確認する

---

## 5. 出力・実装手順（順序厳守）

1. **設定ファイル**
   - `tailwind.config.ts`
   - `next.config.ts`

2. **共通コンポーネント**
   - `app/components/shared/*`

3. **ページ / セクションコンポーネント**
   - `app/components/[page]/*`

4. **ページ本体**
   - `app/page.tsx`
   - 必要に応じて `app/about/page.tsx` など

5. **グローバル設定**
   - `app/layout.tsx`
   - `app/globals.css`（※記述なし、または元コード `<style>` の移植のみ）

6. **アニメーション・インタラクションの接続確認**
   - globals.css にアニメーション CSS がある場合、対応する JavaScript が layout.tsx またはコンポーネントから実行されることを確認
   - 作成したカスタムフック・ユーティリティが全て使用されていることを確認

7. **ビルドエラー確認（必須）**
   - すべての実装完了後、必ず以下を実行する
     ```bash
     npm run build
     ```
   - 以下の観点でエラーが出ないことを確認する
     - TypeScript 型エラーが存在しない
     - Server / Client Component の境界違反がない
     - `next/image`, `next/link` の import 漏れがない
     - 全 `<Image />` に **`width`+`height` または `fill`（＋親の `relative` とサイズ）** があり、`src` に **`/public/` 誤りや存在しないローカルパス**がない
     - 未使用 import / 未定義変数が存在しない
     - Hydration Warning / Error が発生していない
   - ビルドが通らない場合は **実装完了とみなさない**

---

## 7. Tailwind v4 対応ルール（必須）

### バージョン確認（必須）
- `postcss.config.mjs` と `package.json` を確認し、Tailwind のメジャーバージョンを特定すること
- **v4 の場合のみ**、以下のルールを適用する

### Tailwind v4 のときの制約（重要）
- `tailwind.config.ts` へのカラー定義は **期待通り機能しないケースがある**
- そのため **色・影などのデザイントークンは `app/globals.css` の `@theme` に定義する**

### `@theme` の扱い（例外の例外：これは"設定"でありカスタムCSSではない）
- `app/globals.css` への記述が許可されるのは **`@theme` ブロック（カラー・シャドウ等のトークン定義）と、元コード `<style>` の移植のみ**
- `@theme` は **スタイル上書きではなく設定**として扱うため許可

## 原則まとめ（優先順位順）

1. **テキスト内容の完全維持**（最重要）— 元コードの全テキストを一字一句残す。欠落があればビルド成功でも不合格
2. **レイアウト構造の完全維持**（テキストと同等に最重要）— セクションの配置パターン・背景画像・高さ指定・props を変えない。レイアウト変更はビルド成功でも不合格
3. **デザイン・配色の完全維持** — HEXカラー・クラス名を変えない
4. **アニメーション・インタラクションの動作保証** — `opacity: 0` の要素が表示されないまま放置しない。作成したフック/ユーティリティは必ず接続する
5. **Next.js App Router の思想に忠実に分離** — page.tsx は薄く、コンポーネントは厚く
6. **Hydration Error は 1件も許容しない**
7. **`npm run build` が通る状態をゴールとする**
