# サイト構成ルール（プロンプトに含まれる構成・情報設計）

派生版 **`mac-mini-bot-v2`** の **LLM 2 段**索引の一部（主に第 1 段 Gemini マニュアル）。全体は [README](./README.md) / [README](../../README.md)。

## 共通の考え方

- マニュアル手順はプランごとに `config/prompts/basic_lp_manual/`、`basic_cp_manual/`、`standard_cp_manual/`、`advance_cp_manual/` に分割されている。
- 「先方要望があれば他条件より優先」という文言が、構成・原稿系の手順に繰り返し出る。

## BASIC LP（ランディング）

**手順 3（`step_3.txt`）** が全体構成の中核。

- 理論フレーム: ゴールデンサークル、新 PASONA の法則に沿った**セクション一覧・個数・順序**（ライティングはまだ不要）。
- セクションの基本型（例）: メインビジュアル / why（お悩み・解決）/ what（詳細）/ how（ベネフィット）/ 中間オファー / 社会的評価 / さらに詳しい what（料金等）/ フロー / FAQ / 会社概要 / お問い合わせ。
- **中間オファー**: 2 カ所以上。
- **FAQ**: 必ず独立 1 セクション。
- **お問い合わせ**: 必ず独立 1 セクション。
- **会社概要・運営者情報**: 独立 1 セクション、かつお問い合わせの**上**に配置。
- 実店舗系は **Where** セクション必須（会社概要と統合不可）。
- 個人運営は **Who** セクション必須（代表挨拶と統合可）。
- **ヘッダー・フッター禁止**（LP 手順内で繰り返し）。

**手順 4（`step_4.txt`）** でワイヤを「制作可能な原稿＋画像配置説明」まで高解像度化。

- 図解・比較表・Before/After など、AI が生成しうる粒度の指示を詳述。
- テキストと「画像・配置の説明」を明確に分けて出力。
- メインビジュアル、お悩み、中間オファー、実績、FAQ、フォーム、会社概要など、セクションごとの必須要素・文字数制約（例: キャッチ 15 文字以内等）。

**手順 8-2** では `{{STEP_4_OUTPUT}}` を「サイト構成」として **100% 反映**するよう要求（ヘッダーフッター禁止、FAQ アコーディオン必須など UI ルールと併記）。

## BASIC / STANDARD / ADVANCE（コーポレート系）

- **手順 2**（`step_2.txt`）でサイト全体の構成・ページ要否（ブログ行 `{{BLOG_PAGE_LINE}}` 等）を決定。
- **手順 3** 以降でページごとのセクション内容を膨らませ、**手順 7** 前後で Canvas / 最終生成に近い形へ。

**BASIC-CP の手順 7-2（例）** では、`{{STEP_3_OUTPUT}}` を「サイト構成」としてテキスト情報を欠落なく反映することを要求。原稿の厚み・「裏方メモ」と表示テキストの分離もここで指示。

## Manus リファクタ段での構成

`refactoring_instruction_handwork.txt` は Canvas 単一ファイルを解析し、

- **パターン A**: シングル LP → `app/page.tsx` + `app/components/home/*`
- **パターン B**: `useState` による擬似 SPA → `app/about/page.tsx` 等のファイルベースルーティング

へ変換するルールを定義。`page.tsx` は組み立て専用・極薄にする制約が強い。

## 仕様 dict 側（パース後）

`build_basic_lp_spec_raw` 等で生成される YAML メタには `page_structure`・`ux_spec.navigation_model` などが含まれ、**サイト台本**として後続の適用・参照に使われます（詳細は `modules/llm/basic_lp_spec.py` 等）。
