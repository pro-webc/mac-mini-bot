# LLM プロンプトから抽出したルール一覧

`config/prompts/` 配下の複数プロンプト（Gemini 向け `*_manual/*.txt` / Manus 向け `manus/*.txt` / 共通 `common/technical_spec_prompt_block.txt`）に散在するルール・指示を 1 ファイルに集約したもの。

---

## 1. ランタイム入力（プロンプトに差し込まれるデータ）

### 主なプレースホルダ（`{{NAME}}`）

| プレースホルダ | 意味 |
|----------------|------|
| `HEARING_BLOCK` | ヒアリング全文 |
| `STEP_*_OUTPUT` | 前段の応答を次プロンプトに連鎖 |
| `APPO_MEMO` | アポメモ＋営業メモ |
| `EXISTING_SITE_URL_BLOCK` | 既存サイト URL または「記載なし」指示 |
| `HP_COLOR_CLIENT` / `MOOD_CLIENT` | クライアント指定の色・雰囲気（BASIC 系では自動実行時に「ヒアリングを参照せよ」旨に差し替え） |
| `REFERENCE_URL_BLOCK` | 参考サイト URL またはフォールバック文 |
| `BLOG_PAGE_LINE` | STANDARD/ADVANCE でブログ有無を制御する行 |
| `MANUS_REPO_NAME` / `MANUS_REPO_DESCRIPTION` | Manus 向けリポジトリ名・説明 |

### チェーン出力の再利用

- 新規チャットはマニュアル上の「タブ」境界に対応。
- 同一チャット内では `send_message` でコンテキストが保持される場合と、`{{STEP_*}}` で明示埋め込みする場合がある。

## 2. サイト構成ルール

### 共通

- 「先方要望があれば他条件より優先」が構成・原稿系の手順に繰り返し出る。

### BASIC LP

- **手順 3**: ゴールデンサークル・新 PASONA の法則に沿ったセクション一覧・個数・順序。
  - 中間オファー: 2 カ所以上
  - FAQ: 必ず独立 1 セクション
  - お問い合わせ: 必ず独立 1 セクション
  - 会社概要・運営者情報: 独立 1 セクション、お問い合わせの**上**に配置
  - 実店舗系: **Where** セクション必須（会社概要と統合不可）
  - 個人運営: **Who** セクション必須（代表挨拶と統合可）
  - **ヘッダー・フッター禁止**
- **手順 4**: ワイヤを「制作可能な原稿＋画像配置説明」まで高解像度化。テキストと「画像・配置の説明」を明確に分ける。セクションごとの必須要素・文字数制約（例: キャッチ 15 文字以内）。
- **手順 8-2**: `{{STEP_4_OUTPUT}}` を「サイト構成」として **100% 反映**。ヘッダーフッター禁止、FAQ アコーディオン必須。

### BASIC / STANDARD / ADVANCE（コーポレート系）

- **手順 2**: サイト全体の構成・ページ要否を決定。
- **手順 7-2**: 「サイト構成」のテキスト情報を欠落なく反映。「裏方メモ」と表示テキストを分離。

### Manus リファクタ段

- **パターン A**: シングル LP → `app/page.tsx` + `app/components/home/*`
- **パターン B**: `useState` 擬似 SPA → ファイルベースルーティング
- `page.tsx` は組み立て専用・極薄にする。

## 3. デザインルール（配色・レイアウト・UI 品質）

### マニュアル手順内

**BASIC LP**:
- **手順 5**: クライアントの色・雰囲気・参考 URL を前提に配色・フォント・角丸・禁止事項などの「デザイン 3 点セット」を出力。
- **手順 6・7**: ワイヤとデザイン方針を統合した「デザイン指示書」。
- **手順 8-1**: デザイン指示書に基づき 1 枚 LP を生成（ダミーテキスト可、ヘッダーフッター禁止）。
- **手順 8-2**: デザインを保ちつつサイト構成を 100% 反映。画像はプレースホルダで説明（Unsplash 禁止）。ヒーローには必ず画像プレースホルダまたは画像。

**BASIC-CP / STANDARD / ADVANCE**:
- **手順 7-1**: デザイン指示書＋技術要件の箇条書き＋絵文字禁止。
- **手順 7-2**: 手順 6 を「トークン・トーンの正本」、構成案を「文案の正本」とする二層構造。レイアウトはプロとして自律調整可だが**文章量を減らさない**。ヒーローは必ず画像エリア、H1/キャッチは手順 6 のサイズ感より大きく、オーバーレイで可読性確保。装飾の `shadow-*` 禁止。
- 「【画像・配置の指示】」等は公開 UI に出さない。プレースホルダは短く。

### 共通技術ブロック（デザイン系）

- **design_spec の color_scheme / tokens を最優先**。勝手な全面ベタ塗りや派手色の追加は禁止。
- `body`/`html` 背景は低彩度、ブランド色はヘッダー帯・CTA・アクセントに限定。
- テキスト色はパレット内で種類を抑え、コントラスト確保。
- ボタン・リンクの hover/active/focus-visible/disabled を primary 系の明度差で統一（虹色複数 CTA 禁止）。
- タイポは仕様の typography に沿い、h1 はページ 1 つ、長文は可読幅。
- セクション間の縦リズム（`py` スケール）を揃える。
- `bg-[#..]` 任意値は design_spec の色の反映に限定。

### Manus リファクタ段

- デザイン・レイアウト・配色・テキストは **Canvas と同一維持**（文言・数値・カラーコード変更禁止）。
- コンポーネント分割に伴いグリッド・余白・レスポンシブは**プロ品質になるよう調整可**。
- Wire メタの除去のみ可。ユーザー向け原稿の要約・統合は禁止。

## 4. 技術要件（スタック・実装制約）

### スタック

- Next.js App Router, React, Tailwind。
- 1 セクション ≈ 1 コンポーネント。
- `@import` は CSS 最上部。
- 図表はマークアップで実装。

### アイコン

- Lucide / SNS は Simple Icons。
- 存在しないアイコン名禁止。

### 画像

- Unsplash 禁止。
- ヒーローは必ず画像。
- ビジュアル最低 3 意図箇所。
- 最終リファクタで `next/image` + `public/images/`。

### 地図

- 所在地は Google Maps 埋め込み。
- マップエリアに画像プレースホルダ禁止。
- 対応エリア用の装飾画像禁止。

### 装飾

- `shadow-*`（box-shadow）禁止。

### スタイリング

- Tailwind 前提、CSS 変数使用時も要素ごとに色指定、body 継承に依存しない。
- ボタン状態ごとに色指定。

### CSS 実装順序（Manus オーケストレーション）

- `@import` を先頭（コメントの後）
- `:root` / `@theme` / その他の順
- Google Fonts は CSS @import ではなく `layout.tsx` の `<link>`

### Manus リファクタ技術要件

- `next/image`・`next/link` 必須（`<img>` や onClick 遷移禁止）
- Client Component は `"use client"`、ページは原則 Server Component
- 不正な HTML ネスト禁止、`suppressHydrationWarning` on `body`
- Tailwind v4 時は色等を `globals.css` の `@theme` に
- `page.tsx` に className やレイアウト JSX を書かない
- ビルド失敗時にソースを自動修正しない（`npm run build` は検証のみ）

## 5. コンテンツ・コピー・表現

### 絵文字・装飾記号

- 見出し・本文・リスト・表・ボタン・FAQ・フォーム・ナビ・プレースホルダ・装飾的箇条書きに **Unicode 絵文字を含めない**（原稿に明示的にあるものは転記可）。
- `aria-label`・`alt`・`title` にも装飾絵文字を付けない。
- コミットメッセージ・README・モデルのチャット返答にも装飾絵文字を使わない。
- 強調はタイポ・余白・配色・意味のあるアイコンで（アイコンの乱用も避ける）。

### コピー・トーン

- AI らしさを抑えた自然な文体。感情的すぎない。遠回し・冗長・スピリチュアル表現は禁止。
- キラーフレーズの多用禁止（騙される印象を避ける）。
- セクションタイトルは内容が一目で分かる簡潔さ。
- 本文は広告 LP らしく簡潔に。
- お悩み 4〜6 個、解決文には根拠を入れる（「お任せください」だけにしない）。
- フォーム上の返信目安文言（例: 2 日以内にメール連絡）の必須指定。
- プレースホルダ・例文の文体（「ですます」禁止の箇所など）の指定。

### フォーム

- お問い合わせ項目の最低限: 内容選択プルダウン、名前、メール必須、電話、お問い合わせ内容。
- label の視覚的紐付け、`placeholder` は補助のみ、必須はラベル側で分かるように。
- `inputmode` / `autocomplete` は function_spec に従う。

### アクセシビリティ・UX

- タップ領域 **min 44px** 目安（主要リンク・ボタン）。
- **focus-visible** でフォーカスリングを消さない。
- **prefers-reduced-motion** 時は過剰な motion を抑える。
- `<main>` は各ページ 1 つ、`nav` に `aria-label`、見出し階層を飛ばさない。
- モバイルで横スクロールを起こさない。長い表は `overflow-x-auto` またはカード化。

### 「裏方と表示」の分離（CP 手順 7-2）

- 「【テキスト情報】」の見出し・本文・表・ボタン・FAQ のみ表示。
- 制作指示・長い撮影指示は JSX コメントや短い `aria-label` に要約。「【画像プレースホルダー】」等をユーザーに見せない。

## 6. Git / リポジトリ / デプロイ連携

### リポジトリ作成（Manus）

- ワークスペース: **propagate-webcreation**
- **private** リポジトリ、テンプレート **`propagate-webcreation/DefaultSetting`**
- リポジトリ名: `bot-{レコード番号}-{先方名}`。GitHub 規則に合わない文字は `_` / `-` に置換可。
- Description: `test` + 先方名（例: `test株式会社ABC`）
- 同名リポジトリが既に存在する場合は直ちに終了。

### ビルドと push

- `npm run build` が成功するまで修正ループ。
- 成功後に GitHub へ push。
- コミットメッセージ・作業ログに装飾用 Unicode 絵文字を含めない。

### `BOT_DEPLOY_GITHUB_URL`（返答形式）

- push 完了後、返答の**最後の非空行**を次の形式の 1 行だけにする:
  `BOT_DEPLOY_GITHUB_URL: https://github.com/オーナー/リポジトリ.git`
- この行の後に文字・空行・見出しを書かない。
- 報告文・サマリーはすべてこの行より前。
- `.git` 付き URL を推奨。
