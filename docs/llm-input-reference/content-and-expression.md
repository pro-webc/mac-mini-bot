# コンテンツ・コピー・表現・アクセシビリティ

マニュアル各所と `technical_spec_prompt_block.txt` に分散している「文言とユーザー向け品質」のルールを集約します。

## 絵文字・装飾記号

**全プラン共通ブロック**（`technical_spec_prompt_block.txt` の「表現・文字面」）:

- 見出し・本文・リスト・表・ボタン・FAQ・フォーム・ナビ・プレースホルダ・装飾的箇条書きに **Unicode 絵文字を含めない**（原稿に明示的にあるものは転記可）。
- `aria-label`・`alt`・`title` にも装飾絵文字を付けない。
- **コミットメッセージ**・README・**モデルのチャット返答**にも装飾絵文字を使わない。
- 強調はタイポ・余白・配色・意味のあるアイコンで（アイコンの乱用も避ける）。

**Manus 系**: `orchestration_prompt.txt` 先頭、`refactoring_instruction_handwork.txt`、`bot_deploy_instruction.txt` でも同趣旨が繰り返される。

## コピー・トーン（BASIC LP 手順 4 など）

- AI らしさを抑えた自然な文体。感情的すぎない。遠回し・冗長・スピリチュアル表現は禁止。
- キラーフレーズの多用禁止（騙される印象を避ける）。
- セクションタイトルは内容が一目で分かる簡潔さ。
- 本文は広告 LP らしく簡潔に。
- お悩み 4〜6 個、解決文には根拠を入れる（「お任せください」だけにしない）。
- フォーム上の返信目安文言（例: 2 日以内にメール連絡）の必須指定。
- プレースホルダ・例文の文体（「ですます」禁止の箇所など）の指定。

## フォーム（マニュアル＋技術ブロック）

- お問い合わせ項目の最低限（内容選択プルダウン、名前、メール必須、電話、お問い合わせ内容等）は手順 4 で指定。
- `technical_spec_prompt_block.txt` では、**label の視覚的紐付け**、`placeholder` は補助のみ、必須はラベル側で分かるように、`inputmode` / `autocomplete` は function_spec に従う、など。

## アクセシビリティ・UX（技術ブロック）

- タップ領域 **min 44px** 目安（主要リンク・ボタン）。
- **focus-visible** でフォーカスリングを消さない。
- **prefers-reduced-motion** 時は過剰な motion を抑える。
- `<main>` は各ページ 1 つ、`nav` に `aria-label`、見出し階層を飛ばさない。
- モバイルで横スクロールを起こさない。長い表は `overflow-x-auto` またはカード化を検討。

## CP 手順 7-2 にある「裏方と表示」の分離

- 「【テキスト情報】」の見出し・本文・表・ボタン・FAQ のみ表示。
- 制作指示・長い撮影指示は JSX コメントや短い `aria-label` に要約。接頭辞「【画像プレースホルダー】」等をユーザーに見せない。
