# 多段 LLM API で「途中までしか返ってこない」── max_output_tokens 設定漏れの落とし穴

この記事は、**大規模言語モデル（LLM）の API を何度も呼び出して処理を進める仕組み**を作っている人むけです。特定の製品名や業界の前提は不要で、**単体で読める技術メモ**として書いています。

---

## この記事でわかること

- **応答が文章やコードの途中で終わる**現象が、バグではなく **「1 回の返答に許される長さの上限」** に達しただけであることがある、という点。
- その上限は **API を呼び出す側のアプリが設定**しており、多段処理では **後半のステップほど中身が長くなりやすい**、という点。
- だから **「前半はうまくいっている＝全体も安全」ではない**こと、および **対策のチェックリスト**。

---

## まず用語だけ（3 つ）

**多段（マルチターン）パイプライン**  
同じ会話の続きのように、または前の結果を次の入力に貼り付けるようにして、**LLM API を複数回つなぐ**処理の流れ。例としては、「要約 → 箇条書き → JSON 化」の三段、「顧客メモから構成案 → 各章の原稿 → 最後にまとめてコード生成」のような形がある。

**`max_output_tokens`（またはそれに相当するパラメータ）**  
**その 1 回の API 呼び出しで、モデルが返してよい「出力」の長さの上限**（トークン単位）。ここを超えると、モデルは**文の途中やコードの途中で打ち切られる**。プロバイダや SDK によって名前は違うが、考え方は同じである。

**トークン**  
モデルが文章を処理するときのかたまりの単位。**文字数やバイト数と比例しない**。日本語や記号が多いテキスト、インデントの深いソースコードは、見た目の長さよりトークンが多くなりやすい、と覚えておけばよい。

---

## どんな場面で起きるか

次のような流れを想像してください。

1. 最初の数回は、比較的短い返事で済む（構成のたたき台、方針の整理など）。
2. 会話が進むほど、**これまでの出力が次のプロンプトに含まれ**、入力全体が太くなる。
3. 最後のステップでは、「これまでの内容をすべて踏まえて、**長いコードや長いレポートを一気に書いて**」と頼む。

この **3 のような「1 回で大量に書かせるステップ」**で、昔のままの **小さな `max_output_tokens`** を指定していると、**出力が途中で止まる**ことがある。ウェブサイトの複数ページ分をまとめて生成する、長い API 仕様書を一発で出す、といった用途で典型例が出やすい。

---

## 何が起きるか（症状）

次のような **「最後のほうだけおかしい」** 状態になる。

- Markdown のコードブロック（\`\`\`）が閉じていない。
- プログラムの括弧や HTML タグが開いたまま終わる。
- JSON が配列やオブジェクトの途中で終わる。
- ログ上の文字数はそれなりにあるのに、**構造的には完結していない**。

ここで紛らわしいのは、**ストリーミングの不具合**や**ファイル保存時の切り捨て**と症状が似ることである。切り分けの手がかりは、**API が返す「終了理由」（`finish_reason` など）が「長さの上限」**を示しているかどうか、である。

---

## なぜ重大か

### 見た目は「それっぽい」

出力の**先頭から数千トークンまではきちんと読める**ため、ぱっと見は成功に見える。説明文のあとに長いコードが続く形式だと、**説明部分が整っているだけで全体を信じてしまいやすい**。

### 多段では欠損が伝染する

前のステップの出力が次のプロンプトに入る場合、**そこで既に欠けている内容は、後から自動では復活しない**。後続のモデルが足りない部分を推測で埋めると、**事実と違う内容（いわゆる幻覚）**が増える。パーサやビルドは落ちるか、本番で初めて壊れるか、という展開になる。

### エラーにならないことが多い

多くの API は、上限到達時も **HTTP 200 と「途中までの本文」** を返す。アプリ側が終了理由を見ていなければ、**例外も出ず、ジョブは正常完了として記録される**。

---

## 原因の整理（誤解しやすい点）

| よくある思い込み | 実際 |
|------------------|------|
| 「賢いモデルだから、必要な長さだけ返してくれる」 | **返せる長さの上限は、呼び出し側の設定で決まる**。足りなければ途中で止まる。 |
| 「最初のステップは短い返事だから、デフォルトの 8192 で十分」 | **後半だけ**プロンプトも返答も巨大になる。**最初に動いた設定のまま放置**しがち。 |
| 「文字数は十分あるはず」 | **トークン数は文字数とイコールではない**。言語や記号の混ざり方で変わる。 |

SDK のデフォルト値をそのまま使うと、**試作段階では足りても、本番データと本番の最終ステップでは足りない**、というギャップが起きやすい。

---

## 対策（実装チェックリスト）

1. **パイプラインのどのステップで出力が最長になりそうか**を書き出し、その長さに合わせて **`max_output_tokens` を明示的に設定**する。設定値は **環境変数など 1 か所に集約**すると見直しやすい。

2. **終了理由をログに残す**。`MAX_TOKENS` や `length` など、**「長さで打ち切った」**ことが分かるなら、本番では警告やメトリクスに載せる。

3. **結合テストで「末尾が閉じているか」**を見る。コードなら括弧やフェンスの簡易チェック、JSON ならパースを試す。

4. **チームで「長文を出すステップ」をドキュメント化**する。設計レビューのときに「そのステップの上限は十分か」と聞けるようにする。

5. **1 回の応答に詰め込みすぎない**。たとえば「ページごとに API を分ける」「章ごとに生成してから結合する」など、**分割すれば 1 回あたりの出力が短くなる**。

## 事例：ある自動生成プロジェクトで起きたこと（一般化）

以下は、**実際にあった開発**を、製品名を伏せて要約したものである。

**状況**  
業務マニュアルに沿った手順を、チャット形式の API で **何通りか順に実行**するツールだった。入力は顧客ヒアリングのような文章で、途中経過として構成案やページごとの原稿、デザインの指示が出て、**最後のステップで Web サイト用のコンポーネントコードを、複数ページ分まとめて 1 回の応答で出す**、という流れだった。

**問題**  
各ステップの **`max_output_tokens` がデフォルトのまま（例: 8192）** だった。最終ステップでは **1 回の返答が長大になる**ため、**出力がコードの途中で途切れた**。保存ファイルの末尾は、型注釈の途中やタグの半端なところで終わっていた。

**見落とされやすかった点**  
上限を引き上げたのが **最終ステップのデバッグ**だった場合でも、**すでにチャット履歴に記録された「途中までの返答」は、設定を直しただけでは書き換わらない**。さらに調べると、**もっと上流のステップ**では、**複数ページ分の原稿を 1 回で書かせており、そこで先に上限に達していた**。その結果、**構成上は存在するはずのセクションが、原稿の段階で一度も生成されず**、最終コードにも現れない、という欠損が起きていた。

**対応**  
マニュアルに沿った生成処理だけ、共通の **大きめの出力上限**（例として 65536 トークン）を使うようにし、打ち切り時はログで警告するようにした。あわせて、**欠損が出たステップからパイプラインをやり直す**必要があることを運用で共有した。

**教訓**  
これはライブラリのバグではなく、**設定とデータ量のミスマッチ**に近い。とはいえ結果として **品質と信頼性を大きく損なう**。

## 補足：開発を AI アシスタントと進めたときの人間側の罠

実装を **エディタ上の AI やコーディング支援ツールと一緒に進める**と、次のことが起きやすい。

- 新しい API 呼び出しやプロンプトを足すたびに **`generation_config` は触らず**、**「とりあえず動いた」短いステップ**だけで満足してしまう。
- **後半のステップだけ**プロンプトと履歴が太り、**そこで初めて**上限問題が顔を出す。そのため原因を **モデルの性能やプロンプトの書き方**に求めがちになる。
- 人間の確認も **最後に開いた成果物の末尾**に集中し、**上流の応答ファイルが既に切れている**ことに気づかない。

だから、**「出力上限はいくつか」「打ち切り時にログを出すか」**を、機能追加のチェックリストに**明示的に入れる**のが効果的である。技術的対策と、プロセス上の対策はセットである。

---

## まとめ

- LLM API を**何度もつなぐ**と、**後半の 1 回あたりの入出力が太くなりやすい**。
- **`max_output_tokens` が足りないと、エラーにならずに途中切れ**することがある。
- **終了理由の観測**、**長文ステップの特定**、**必要ならステップ分割と再実行**をセットで設計すると安全である。
